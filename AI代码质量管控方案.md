# AI 生成代码质量管控机制（增强版）

> **版本**：V2.0 | **适用范围**：所有使用 AI 辅助编码工具（Copilot/Cursor/ChatGPT 等）的研发团队
> **核心原则**：AI 代码进入测试前，必须经过人类工程师的批准。没有 CR，没有测试。

---

## 一、背景与目标

### 问题现状

随着 AI 编程工具的普及，开发效率大幅提升，但随之而来的问题正在显现：

- AI 生成的代码**逻辑正确率虎头蛇尾**，边界条件、异常处理尤为薄弱
- 开发人员对 AI 输出缺乏审辨，直接提测，把**测试当作唯一兜底层**
- 线上故障溯源困难，责任边界模糊，测试团队长期处于被动背锅状态

### 目标

| 目标 | 衡量指标 |
|------|---------|
| 开发对 AI 代码质量负主责 | 缺陷逃逸率（线上/测试阶段发现缺陷比）下降 ≥30% |
| 低质量代码在提测前被拦截 | 无效提测次数归零 |
| 测试团队摆脱基础逻辑兜底职责 | 测试资源聚焦在业务验证，而非代码正确性 |

---

## 二、核心机制：四道强制门禁

> **门禁原则**：任意一道门禁未通过，提测流程自动阻断，测试团队无需协商，直接拒绝。

---

### 门禁一：CI 质量红线（自动化强制）

在 CI/CD 流水线中增加质量门禁步骤，**门禁未过则提测流水线直接失败**，测试人员看不到可测版本。

| 检查项 | 阈值 | 备注 |
|--------|------|------|
| 单元测试（新增/修改代码） | 100% 通过 | 任何失败均阻断 |
| **增量代码覆盖率**（本次 PR 新增/修改部分） | ≥ 70% | 采用**增量**而非整体，避免存量代码欠债造成阻力 |
| 静态代码扫描（Sonar/ESLint/Checkstyle 等） | 无新增**阻断性**问题（Blocker/Critical） | Warning 级别不阻断但需在 PR 说明中解释 |
| 流水线冒烟测试 | 100% 通过 | 冒烟用例由**开发负责编写和维护**，测试负责验收 |

> [!IMPORTANT]
> **覆盖率策略说明**：使用增量覆盖率（Diff Coverage）而非整体覆盖率，可通过 `diff-cover`、`Jacoco` 的 PR 增量报告等工具实现。这是本方案与原版的关键改进点。

**CI 配置示例（GitLab CI）：**
```yaml
quality-gate:
  stage: quality
  script:
    - run_unit_tests.sh
    - check_diff_coverage.sh --threshold 70
    - run_sonar_scan.sh --fail-on-blocker
    - run_smoke_tests.sh
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
  allow_failure: false  # 关键：门禁失败则整个流水线失败
```

### 门禁二：AI 代码声明 + 人工 Code Review

#### 2.1 AI 代码声明（开发自主申报）

> [!NOTE]
> **申报荣誉机制**：如实申报 AI 辅助不受额外审查负担加重，隐瞒 AI 使用且线上故障时，责任加重处理。鼓励如实填写，不惩罚透明。

开发在提交 PR 时，必须在 PR 描述中填写：

```markdown
## AI 辅助声明（必填）
- [ ] 本 PR 未使用 AI 辅助
- [ ] 本 PR 使用了 AI 辅助，AI 生成代码占比约：____%
  - 使用工具：（如 Copilot / Cursor / ChatGPT / 其他）
  - AI 生成的主要模块：___________
```

**触发强制 CR 的条件**：AI 辅助代码比例 > 0%，或连续 ≥ 10 行由 AI 生成。

#### 2.2 Code Review 规则

- **审查人要求**：至少 1 名非作者的**资深工程师**（工作年限 ≥ 3 年或 Senior 级别以上）
- **形式主义防范**：Reviewer **必须留至少 1 条实质性评论**（非 "LGTM"、"好的"、"OK" 等），否则视为无效 CR
- **时限要求**：CR 应在提测申请后 **24 小时内**完成，超时升级至 TL 处理

#### 2.3 AI Code Review Checklist（PR 模板必填，全部勾选方可合并）

```markdown
## AI Code Review Checklist（Reviewer 必须逐项确认）

### 逻辑正确性
- [ ] 边界条件已覆盖（空值、零值、最大值、并发场景）
- [ ] 异常处理完整（不吞异常、有合理的降级或告警）
- [ ] 业务逻辑与需求文档一致，无过度实现或缺失实现

### 性能与安全
- [ ] 无循环内数据库查询（N+1 问题）
- [ ] 无明显内存泄漏风险（大对象、未关闭资源等）
- [ ] 无硬编码敏感信息（密码、Token、AK/SK 等）
- [ ] 输入数据有校验，无 SQL 注入 / XSS 等基础安全漏洞

### 代码质量
- [ ] 无死代码或冗余逻辑
- [ ] 命名清晰，无魔法数字
- [ ] 单元测试覆盖了主要逻辑路径和异常分支
- [ ] 无明显的 AI 幻觉痕迹（如调用不存在的方法、错误引用第三方 API）

**Reviewer 签名**：________________  **日期**：________________
**实质性评论链接**（必填，粘贴评论 URL）：________________
```

### 门禁三：测试准入声明

测试人员在启动测试前，要求开发提供以下三项材料，**缺任何一项直接退回，不排期，不计入测试绩效**：

| 材料 | 形式 | 说明 |
|------|------|------|
| ① CI 通过证明 | 流水线截图或链接 | 显示所有质量门禁绿色通过 |
| ② CR 批准链接 | PR 链接 | 可见 Checklist 全部勾选 + 实质性评论 |
| ③ AI 代码自检声明 | 勾选表（见下） | 开发签字确认 |

**《AI 代码自检声明》表格：**

```
项目/迭代名称：___________   提测日期：___________
开发姓名：___________

本人确认以下各项：
☐ 本次提测代码已通过 CI 全部质量门禁
☐ 所有 AI 辅助代码已完成 CR，Checklist 全部勾选
☐ 已对 AI 生成的边界条件和异常逻辑进行人工验证
☐ 本人已在本地完成基础功能自测，主流程可正常运行
☐ 无已知的阻断性缺陷被刻意隐瞒

开发签名：___________   日期：___________
```

### 门禁四：故障追溯倒查（事后问责链）

> [!TIP]
> 这是本方案新增的关键机制，让门禁不只是"防火墙"，还具备**事后问责能力**，大幅提升严肃性。

每次**线上故障复盘**，必须自动拉取以下信息作为复盘必看项：

- 对应 PR 的 CI 流水线状态记录
- CR 记录（Reviewer 是谁、留了什么评论、何时批准）
- AI 代码自检声明副本
- 发布记录与审批链

**故障定责原则**：
- 门禁齐全但仍出现故障 → 开发承担主要责任，测试免责
- 绕过门禁（如 CR 走形式、CI 未通过强推）→ 开发 + CR 人共同承担，加重处理
- 门禁流程本身存在漏洞导致故障 → 由质量负责人承担流程责任

## 三、角色与责任边界

| 角色 | 职责 | 不得推卸的边界 | 违规后果 |
|------|------|--------------|---------|
| **开发** | 保证 AI 代码通过 CI 门禁 + 完成 CR + 提交自检声明 | 不得以"AI 生成的"为由免责 | 返工，开发自测修复；绩效扣分 |
| **CR 人（Reviewer）** | 逐项核查 AI Code Review Checklist，留实质性评论 | 形式主义点 Approve 视为失职 | 线上故障连带责任，记录在案 |
| **测试** | 仅对满足所有门禁的版本执行测试，不负责基础逻辑兜底 | 不得因"开发压力"接受违规提测 | 因流程缺失导致的质量问题，测试**完全免责** |
| **项目经理/TL** | 背书并执行新规则，不得以"进度压力"要求测试绕过门禁 | 门禁是硬性规定，不存在"特事特办" | 绕过门禁导致的故障，管理层承担流程责任 |

> [!CAUTION]
> **管理层背书是最关键的成功因素**。技术门禁靠工具强制，流程门禁（如测试拒绝接测）需要管理层公开支持。建议在方案宣导时，由研发负责人和项目负责人**联署发布**，明确声明"测试打回违规提测是职责履行，不是不配合"。

## 四、落地步骤（3 周计划）

### 第 1 周：技术建设

- [ ] **Day 1-2**：在 CI 流水线中集成增量覆盖率检查（推荐工具：`diff-cover` / Jacoco PR 报告）
- [ ] **Day 2-3**：配置静态扫描门禁（SonarQube 或 ESLint，设置 Blocker 级别失败阻断）
- [ ] **Day 3-4**：配置 Git 分支保护规则（GitLab/GitHub：merge 前需 CI 通过 + ≥1 个有效批准）
- [ ] **Day 4-5**：制作 PR 模板（含 AI 声明字段 + AI Code Review Checklist）
- [ ] **Day 5**：在 Wiki 或提测系统发布《AI 代码自检声明》表格模板

### 第 2 周：管理准备与宣导

- [ ] 获取研发负责人 + 项目负责人**联署认可**，形成书面通知
- [ ] 召开团队全员会，宣布新规则，重点说明：
  - 门禁是硬性规定，无例外
  - 测试打回违规提测是合理职责
  - 如实申报 AI 使用不受额外惩罚
- [ ] 在团队可见位置张贴规则摘要（物理看板或飞书/钉钉置顶公告）

### 第 3 周：试运行与校准

- [ ] 选择 1 个迭代严格试运行，测试团队执行打回规则
- [ ] 收集数据：违规次数、违规类型（哪道门禁最多未过）
- [ ] 第 3 周末召开复盘会，根据实际情况微调阈值（如覆盖率可从 60% 起步）
- [ ] 正式发布 V1.0 规则

---

## 五、违规处理（递进原则）

| 违规次数 | 处理措施 |
|---------|---------|
| **第 1 次** | 测试打回，发送邮件至开发本人及其直属 Leader，说明违规原因 |
| **第 2 次** | 记录为"无效提测"，纳入开发绩效评估，当月 KPI 受影响 |
| **第 3 次** | 暂停该开发的提测权限（需重新通过质量培训后恢复），上报部门负责人 |
| **CR 人走形式** | 线上故障追溯时连带定责，同步记录至 Reviewer 绩效档案 |
| **管理层强推绕过** | 故障复盘中明确记录，纳入管理层问责范围 |

## 六、常见阻力与应对话术

| 阻力场景 | 应对方式 |
|---------|---------|
| "进度紧，这次先特事特办" | "门禁是流程边界，特事特办的代价是线上故障。延期提测比线上故障代价小。" |
| "我的代码 AI 只生成了一小部分" | "如实填写比例即可，不影响流程。比例低则 CR 压力也小。" |
| "单元测试太难写" | "写不出单元测试通常说明代码耦合度过高，这是需要修复的设计问题，而非免责理由。" |
| "测试这样做会影响团队关系" | "管理层已背书。测试打回是职责，不是个人行为，有管理层承担后续沟通。" |

---

## 七、一句话总结

> **"AI 代码进入测试前的最后一步，必须有人类工程师的批准。没有 CR，没有测试。"**

这个方案不依赖复杂工具链，核心是**改变流程 + 明确责任 + 敢于拒绝**。
成功的关键不在于技术，而在于**管理层是否真正站出来背书，并在第一次被绕过时坚决执行**。
