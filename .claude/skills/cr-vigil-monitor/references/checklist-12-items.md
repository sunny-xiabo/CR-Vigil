# AI Code Review Checklist (12 Items)

本文件将 AI Code Review Checklist 结构化为 12 个可验证的检查项，每项带唯一 ID。Reviewer 必须逐项确认，全部勾选方可合并。

---

## 一、逻辑正确性 (Logic Correctness)

### CK-01: 边界条件已覆盖

检查要点:
- 空值 (null / undefined / 空字符串 / 空数组 / 空对象)
- 零值 (0 / 0.0 / 空集合)
- 最大值 (数组边界 / 数值上限 / 字符串长度上限)
- 并发场景 (竞态条件 / 共享状态)

判定标准: 代码是否对所有上述边界场景有显式处理或有合理的不需要处理的理由。

### CK-02: 异常处理完整

检查要点:
- 异常被捕获后是否有处理逻辑（不吞异常）
- 是否有合理的降级策略
- 是否有适当的告警或日志记录
- 第三方调用的异常是否被正确处理

判定标准: 所有 try-catch 块均有实际处理逻辑，异常不会被静默吞掉。不存在空的 catch 块。

### CK-03: 业务逻辑与需求文档一致

检查要点:
- 实现与 PR 描述的需求一致
- 无过度实现（超出需求的额外功能）
- 无缺失实现（需求中明确但代码中缺失）
- 业务规则计算方式与规格一致

判定标准: 代码实现严格匹配需求描述，不多不少。

---

## 二、性能与安全 (Performance & Security)

### CK-04: 无循环内数据库查询 (N+1 问题)

检查要点:
- for/while/forEach 循环内是否有数据库查询
- 是否可通过批量查询替代逐条查询
- ORM 的 lazy loading 是否被正确使用

判定标准: 循环体内不存在独立的数据库访问调用。如有批量操作，使用了批量接口。

### CK-05: 无明显内存泄漏风险

检查要点:
- 大对象是否及时释放
- 资源（文件句柄、数据库连接、网络连接）是否正确关闭
- 事件监听器是否在组件销毁时移除
- 缓存是否有大小限制或过期策略

判定标准: 所有资源有明确的关闭/释放路径，无无限增长的数据结构。

### CK-06: 无硬编码敏感信息

检查要点:
- 不存在硬编码的密码
- 不存在硬编码的 Token / Access Key / Secret Key
- 不存在硬编码的 API Key
- 不存在硬编码的证书或私钥

判定标准: 所有敏感信息通过环境变量、配置中心或 Secrets Manager 获取。

### CK-07: 输入数据有校验

检查要点:
- 用户输入是否经过校验（类型、长度、格式、范围）
- 是否存在 SQL 注入风险（参数化查询）
- 是否存在 XSS 风险（输出编码）
- 文件上传是否有类型和大小限制

判定标准: 所有外部输入有显式校验逻辑，数据库操作使用参数化查询。

---

## 三、代码质量 (Code Quality)

### CK-08: 无死代码或冗余逻辑

检查要点:
- 无未被调用的函数或方法
- 无被注释掉的大段代码
- 无重复的逻辑块（可通过抽取公共方法消除）
- 无永远不会被执行到的分支

判定标准: 代码中不存在永远不会执行或永远不产生效果的语句。

### CK-09: 命名清晰，无魔法数字

检查要点:
- 变量名、函数名能准确表达其用途
- 不存在未命名的数值常量（魔法数字）
- 枚举值有明确的名称和含义
- 缩写是通用的、可理解的

判定标准: 任何人在不阅读上下文的情况下，能通过命名理解变量或函数的用途。

### CK-10: 单元测试覆盖了主要逻辑路径和异常分支

检查要点:
- 正常路径（happy path）有测试
- 异常路径（error path）有测试
- 边界条件有测试
- 测试用例有断言（非仅执行无验证）

判定标准: 新增/修改的代码有对应的测试用例，且测试验证了实际输出。

### CK-11: 无明显 AI 幻觉痕迹

检查要点:
- 调用的方法/函数是否存在（非 AI 臆造）
- 引用的第三方 API 是否正确（API 名称、参数签名）
- 引用的库版本是否真实存在
- 逻辑是否存在明显的不合理或自相矛盾

判定标准: 代码中涉及的所有方法、API、库均由实际代码或文档支撑，不存在虚构引用。

### CK-12: 代码符合团队编码规范

检查要点:
- 缩进、空格、换行符合团队规范
- 文件名、目录结构符合项目约定
- 注释准确且有价值（非废话注释）
- 错误消息清晰且有意义

判定标准: 代码风格与项目现有代码一致，格式工具（如 ESLint/Prettier/Black）无报错。

## 形式主义检测补充

当 Reviewer 提交的评论符合以下任一模式时，对应 Checklist 项的勾选需要重新审查:
- 评论仅含 "LGTM" / "looks good" / "好的" / "没问题" / "OK" / "+1" / "Approved" / "通过" 且无其他内容
- 评论字数 < 10 且无代码引用
- 评论与 Checklist 具体项无任何对应关系
